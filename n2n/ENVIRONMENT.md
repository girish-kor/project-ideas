# ENVIRONMENT.md — N2N Toolz Environment Variables Reference

---

## 1. App-Level Environment Variables (N2N Toolz IDE)

These control the behavior of the N2N Toolz application itself.

| Variable | Type | Default | Description |
|---|---|---|---|
| `N2N_WORKSPACE_DIR` | string | `~/n2n-toolz-workspace` | Root workspace directory |
| `N2N_PLUGIN_DIR` | string | `~/.n2n-toolz/plugins` | Plugin installation directory |
| `N2N_LOG_LEVEL` | string | `info` | App log level: `debug\|info\|warn\|error` |
| `N2N_LOG_DIR` | string | `~/.n2n-toolz/logs` | Log file directory |
| `N2N_MAX_HISTORY` | number | `100` | Max undo/redo history entries |
| `N2N_PLUGIN_REGISTRY` | string | `https://registry.n2ntoolz.io/api/v1` | Plugin registry URL |
| `N2N_SANDBOX_MODE` | boolean | `true` | Enable vm2 sandbox for embedded runtime |
| `N2N_MAX_NODES` | number | `10000` | Max nodes per workflow canvas |
| `N2N_DEFAULT_TIMEOUT` | number | `30000` | Default node execution timeout (ms) |
| `N2N_AUTO_SAVE_MS` | number | `5000` | Auto-save debounce interval (ms, 0=off) |
| `NODE_ENV` | string | `development` | `development\|production` |

## 2. Generated Application Environment Variables

These are used by the **Node.js code N2N Toolz generates**, not the IDE itself.

### General

| Variable | Description | Example |
|---|---|---|
| `PORT` | HTTP server port | `3000` |
| `HOST` | HTTP server host | `0.0.0.0` |
| `NODE_ENV` | Runtime environment | `production` |
| `LOG_LEVEL` | Application log level | `info` |

### Database

| Variable | Node Type | Description |
|---|---|---|
| `DATABASE_URL` | mysql-query / postgres-query | Full connection string |
| `MYSQL_HOST` | mysql-query | MySQL hostname |
| `MYSQL_PORT` | mysql-query | MySQL port (default: 3306) |

## 3. .env File Hierarchy

N2N Toolz loads env vars in this priority (highest first):

```
process.env (system environment)
    ↑
env/.env.prod     (if NODE_ENV=production)
    ↑
env/.env.dev      (if NODE_ENV=development)
    ↑
env/.env          (base defaults)
```

## 4. Auto-Generated .env.example

N2N Toolz generates a `.env.example` file at export time by scanning all `secret: true` config fields:

```bash
# Auto-generated by N2N Toolz v1.0.0
# Copy this file to .env and fill in the values

# ── Database ──────────────────────────────────────
# MySQL connection URL (REQUIRED)
MYSQL_QUERY_CONNECTION=

# ── Auth ──────────────────────────────────────────
# JWT signing secret (REQUIRED)
JWT_SIGN_SECRET=

# Token expiry duration
JWT_SIGN_EXPIRES_IN=1h

# ── Messaging ─────────────────────────────────────
# SMTP server credentials (REQUIRED)
SMTP_EMAIL_USER=
SMTP_EMAIL_PASS=
```

## 5. Env Variable Resolution in Config Fields

When a node config field has `secret: true`, the value stored in the project file is a reference string:

```json
{
  "config": {
    "connection": "env:DATABASE_URL",
    "apiKey":     "env:MY_API_KEY"
  }
}
```

At runtime, `env:DATABASE_URL` is resolved to `process.env.DATABASE_URL`.

In generated code:
```javascript
const connection = process.env.DATABASE_URL;
const apiKey     = process.env.MY_API_KEY;
```

## 6. Env Validation at Runtime (Generated Code)

```javascript
// Auto-injected at top of generated index.js when required vars exist
const REQUIRED_ENV = ['DATABASE_URL', 'JWT_SECRET'];
const missing = REQUIRED_ENV.filter(k => !process.env[k]);
if (missing.length > 0) {
  console.error(`[N2N] Missing required environment variables: ${missing.join(', ')}`);
  process.exit(1);
}
```
