# GLOSSARY.md — N2N Toolz Terms and Definitions

---

| Term | Definition |
|---|---|
| **Node** | A visual block representing a single operation in a workflow (e.g., HTTP Request, JSON Parse). Every node has input ports, output ports, and a configuration form. |
| **Edge** | A connection (line) drawn between an output port of one node and an input port of another. Represents data flow. |
| **Port** | A connection point on a node. Input ports (left side) receive data; output ports (right side) emit data. Each port has a data type. |
| **Workflow** | A directed acyclic graph (DAG) of nodes and edges that defines a complete process, stored as a `.n2nflow` file. |
| **Graph** | The in-memory representation of all nodes and edges in a workflow. Managed by `graphStore`. |
| **Canvas** | The infinite 2D workspace where nodes are placed, connected, and arranged visually. |
| **Node Palette** | The left sidebar panel listing all available node types organized by category. |
| **Properties Panel** | The right sidebar panel showing the configuration form for the currently selected node or edge. |
| **Config Schema** | The definition of a node's configuration fields (labels, types, defaults, validation rules). Drives the Properties Panel form. |
| **Node Definition** | The full contract for a node type: identity, ports, config schema, execute function, and code generation function. |
| **Node Registry** | The central in-memory registry of all available node types (core + plugins). |
| **Plugin** | An npm package that exports one or more custom node definitions. Installed into N2N Toolz and appears in the Node Palette. |
| **Code Generation** | The process of converting a workflow graph (JSON) into human-readable, production-ready Node.js source files. |
| **AST** | Abstract Syntax Tree — the intermediate tree structure used by the Code Generation engine before emitting JavaScript source text. |
| **Scope Map** | A map of variable names generated by `ScopeResolver` that links each node's output to a JavaScript variable name in the generated code. |
| **Topological Sort** | An ordering algorithm that arranges nodes so each node is processed after all its upstream dependencies. Used to determine execution order. |
| **Execution Context** | An object injected into every node's `execute()` function containing: env vars, logger, abort controller, and workflow state. |
| **Embedded Runtime** | The live execution engine inside N2N Toolz that runs workflow nodes directly (without generating code first), using vm2 for sandboxing. |
| **Sandbox Mode** | A security constraint mode where nodes run inside an isolated `vm2` environment with restricted access to `fs`, `child_process`, and `net` modules. |
| **Breakpoint** | A debug marker on a node that pauses execution when that node is about to execute. Supports optional condition expressions. |
| **Watch Expression** | A JavaScript expression monitored in the Variables tab during a debug session. Its value is evaluated after each node executes. |
| **Trigger Node** | A node that starts a workflow. Has no input ports. Examples: HTTP Trigger, Cron Trigger, Manual Trigger. |
| **Terminal Node** | A node that ends a workflow branch. Has no output ports. Examples: HTTP Response, Return Value, Throw Error. |
| **Sub-Workflow** | A complete workflow embedded as a single node inside a parent workflow. Generates a reusable async function. |
| **Fan-Out** | When one output port connects to multiple downstream nodes, causing them to execute in parallel. |
| **Merge** | A node that waits for multiple parallel branches to complete and combines their results. |
| **Hot Reload** | The automatic rebuild and restart of the dev server when a workflow is saved. |
| **Incremental Export** | A code generation optimization that only rebuilds files for workflows that changed since the last export. |
| **n2nflow** | The file format for a single workflow graph (JSON). Extension: `.n2nflow`. |
| **n2n project** | A folder-based project bundle containing all workflows, env files, and settings. Extension: `.n2n`. |
| **IPC** | Inter-Process Communication — the channel Electron uses for the Renderer (React UI) to call the Main Process (Node.js). |
| **Secret Field** | A config field marked `secret: true`. The value is masked in UI, encrypted in the project file, and rendered as `process.env.VAR_NAME` in generated code. |
| **DataType** | The declared type of data flowing through a port: `any`, `string`, `number`, `boolean`, `object`, `array`, `stream`, `buffer`, `error`. |
| **Run ID** | A unique UUID assigned to each workflow execution, used to track logs, outputs, and debug state per run. |
| **Run Record** | A stored summary of a past workflow execution including status, duration, and input. |
| **Dependency Resolver** | The component that maps node types to their required npm packages to generate a `package.json`. |
| **Prettier** | The code formatter applied to generated Node.js source files to ensure consistent style. |
| **CJS** | CommonJS module system (`require()` / `module.exports`). One of two output modes for code generation. |
| **ESM** | ECMAScript Modules (`import` / `export`). The default and preferred output mode for code generation. |
| **vm2** | A third-party Node.js library for running code in a secure, isolated virtual machine context. Used for Sandbox Mode. |
| **Zustand** | The React state management library used in N2N Toolz. Provides lightweight global stores with hooks. |
| **Immer** | A library that enables direct mutation syntax in Zustand store reducers, simplifying complex state updates. |
| **React Flow** | The canvas library used to render and manage the node graph visually. |
| **recast** | A JavaScript AST toolkit used by the Code Generation engine to build and print AST nodes as source code. |
| **Node.js** | The JavaScript runtime that N2N Toolz generates code for. All exported projects are Node.js applications. |
